## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GraphQL API → SPA DOM | Server-provided vulnerability data (title, primaryUrl, versions) and the echoed repoUrl are rendered into the browser DOM |
| User input → client validation | The URL field's client-side validation is UX feedback, not a security control |
| npm registry → build | New Vite/React/urql/Tailwind packages enter the workspace dependency graph |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-06 | Info Disclosure / XSS | Rendering server vulnerability fields + repoUrl into the DOM | medium | mitigate | React escapes text nodes by default; render all vulnerability fields as inert text; use primaryUrl only as an href with `rel="noopener noreferrer" target="_blank"`; never use dangerouslySetInnerHTML. |
| T-06-07 | Tampering | Treating client-side URL validation as a security control | low | accept | Client validation is cosmetic UX only; the authoritative fail-closed guard is server-side `parseGithubUrl` in the resolver (Plan 01, T-06-01). Documented as defense-in-depth, not the boundary. |
| T-06-SC | Tampering | npm installs (vite, react, react-dom, urql, @urql/core, tailwindcss, @tailwindcss/vite) | high | mitigate | Install verbatim pinned versions from RESEARCH Standard Stack; graphql pinned to 16.14.2 (root override too). RESEARCH Package Legitimacy Audit lists all as Approved (SUS = "too-new" false positives on mature, high-download, official-repo packages; no [SLOP], no [ASSUMED], no postinstall scripts) — no human legitimacy checkpoint required; pins applied exactly. |
</threat_model>
