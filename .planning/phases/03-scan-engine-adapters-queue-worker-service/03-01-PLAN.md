<objective>
Establish the Phase 3 contracts and durable queue/state seam before adapters and worker lifecycle code are added. Replace the Phase 1 in-memory ScanStore with Redis-backed domain state while preserving the shared two-entrypoint topology.

Purpose: Every later adapter and worker operation must compile against explicit ports, and Phase 4 must be able to consume ScanService without transport-specific behavior.
Output: Approved dependency pins, typed queue/repository contracts, Redis repository, ScanService, shared module wiring, and focused contract tests.
</objective>

<scope_rationale>
This plan is a cohesive contracts-and-state boundary: dependency pins, typed queue/repository interfaces, Redis persistence, ScanService, and the shared module are the producer/state seams that later adapter and worker plans consume. Task 1 establishes the approved package surface, Task 2 establishes durable repository behavior, and Task 3 wires the service and shared module only after those contracts exist; keeping this sequence preserves single-file ownership and makes each task executable without mixing subprocess or worker lifecycle concerns into the state boundary.
</scope_rationale>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| API/service → BullMQ/Redis | Queue payload and domain state cross into a shared datastore. |
| Worker/repository → Redis | Untrusted or late job activity can attempt state mutation. |
| Configuration → Redis client | Validated host, port, and temporary-root settings control infrastructure connections. |

## ASVS Level 1 Controls

Apply OWASP ASVS Level 1 controls to every boundary in this plan: V5/V14 require typed input and fail-closed configuration validation; V5/V8 require discrete subprocess argv, `shell:false`, and confinement of generated paths beneath `SCAN_TMP_DIR`; V7/V8 require bounded, redacted error details and exclusion of raw credentials, Redis diagnostics, and uncontrolled paths from persisted state; V12/V14 require bounded Redis operations, seven-day TTL refreshes on both keys, finite producer retry settings, and explicit handling of missing Redis records without unbounded memory or retry behavior.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|---|---|---|---|---|---|
| T-03-01 | Tampering | ScanService queue payload | high | mitigate | Use a single typed `{scanId, repoUrl}` payload and no paths, credentials, or report contents; keep producer and processor on one queue name. |
| T-03-02 | Tampering | ScanRepository terminal transitions | high | mitigate | Use WATCH/MULTI/EXEC with conflict retry, or an equivalent Lua check-and-mutate script, to guard terminal status atomically; refresh seven-day expiry on both metadata hash and ordered vulnerability list in the same mutation. |
| T-03-03 | Availability | Redis retention and BullMQ connections | medium | mitigate | Refresh TTL on every write, use no ioredis keyPrefix, and configure finite producer retries separately from worker blocking connection settings. |
| T-03-04 | Information disclosure | Scan error field | medium | mitigate | Persist only bounded structured failure data; keep raw Redis/subprocess diagnostics outside the domain record. |
| T-03-SC | Tampering | npm dependency installation | high | mitigate | Human legitimacy checkpoint before installing BullMQ and exact package pins; no package install precedes approval. |
</threat_model>
