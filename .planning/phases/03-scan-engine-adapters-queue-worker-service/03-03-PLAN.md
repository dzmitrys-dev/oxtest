<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| BullMQ job → WorkerHost | Typed job data controls the worker's repository target and generated work identifiers. |
| Worker → Redis repository | Status, ordered findings, and bounded failure reasons become shared state. |
| Configuration → worker bootstrap | Fault and readiness settings control adapter construction and process assertions. |

## ASVS Level 1 Controls

- **Input and configuration validation:** Validate the typed BullMQ job payload, adapter-fault allowlist, readiness marker, Redis settings, and worker environment fail-closed; named test faults are rejected outside test mode.
- **argv, shell, and path safety:** Keep worker and process-contract spawns on discrete argv arrays with `shell:false`; accept only generated scan paths from adapter ports and do not let job data select arbitrary executable arguments or filesystem roots.
- **Bounded and redacted errors:** Persist only the bounded/redacted `ScanError` category and detail, cap process-contract diagnostics and timeouts, and exclude credentials, raw stderr, Redis internals, and uncontrolled paths from Redis state and readiness output.
- **Availability, Redis, and memory risks:** Enforce concurrency 1, await parser appends and cleanup, bound readiness/polling timeouts, use worker Redis blocking retry settings only where required, and avoid report buffering or unbounded retry/memory growth during Redis or cleanup failures.
