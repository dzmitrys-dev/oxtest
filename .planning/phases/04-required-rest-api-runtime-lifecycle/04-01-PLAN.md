<objective>
Expose the already-built async scan engine over the REQUIRED REST surface through the single shared `ScanService`: `POST /api/scan`, `GET /api/scan/:scanId`, and `GET /health`. Controllers are thin transport adapters (ARCH-01) — all validation and status mapping live in the transport layer; `ScanService` stays free of URL parsing and HTTP concerns.

Purpose: Deliver success criteria #1, #2, #3 (ROADMAP Phase 4) — clients can submit and poll scans over REST, with a Redis-backed health probe. This is the first public HTTP attack surface, so the GitHub-URL input boundary (D-01/D-02/D-03) is the primary security control.

Output: A `github-url` validator + pipe, request/response DTOs, a thin `ScanController`, a `HealthService`/`HealthController`, and their wiring into `AppModule` — proven by Jest specs that NEVER import the `@nestjs/bullmq`-wired module graph (the confirmed `@swc/core` miette panic; STATE.md).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (`POST /api/scan` body) | Untrusted repo URL crosses here — the first public HTTP attack surface; SSRF / command-injection / look-alike-host source. |
| client → API (`GET /api/scan/:scanId`) | Untrusted scanId path param; response must not over-expose the domain object. |
| client → API (`GET /health`) | Unauthenticated liveness probe; response must not leak internal detail. |
| API → Redis (`REDIS_CLIENT` PING) | Reuses an existing trusted connection; no new credentials or connection introduced. |
